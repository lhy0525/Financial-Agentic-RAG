# 简历编写原则（AI / RAG / Agent 项目方向）

## 1. 四段式项目结构

项目描述优先采用“背景 -> 目标 -> 过程 -> 结果”的四段式，让招聘方快速理解项目为什么做、你要解决什么、你如何落地、最终带来什么价值。

- **背景**：说明项目所在场景，例如本地优先的金融问答、招股书资料分析、投资研究辅助、企业知识库问答等。
- **目标**：说明技术目标或业务目标，例如提升检索可解释性、降低幻觉风险、支持 PDF/TXT 上传索引、构建标准化 MCP 工具服务。
- **过程**：写清个人贡献、技术方案和工程细节，例如 SQL-first evidence path、证据合并与校验、RAG 检索链路、Agent 工具调用、前后端 Demo 联调。
- **结果**：用真实结果或建议量化点收尾，例如检索命中率、响应耗时、索引文档规模、测试覆盖、交付周期等。

## 2. Bullet 写法

每条 bullet 推荐使用“动作 + 技术方案 + 工程细节 + 结果”的结构，避免只有工具堆叠或职责描述。

推荐动词：

- 设计
- 实现
- 构建
- 优化
- 集成
- 主导
- 建立

避免使用：

- 参与
- 协助
- 了解
- 负责相关工作

示例结构：

> 设计金融 RAG 问答链路，基于 SQL-first evidence path 整合结构化财务数据与招股书文本证据，增加来源追踪与证据校验机制，建议量化检索准确率、引用覆盖率和响应耗时。

## 3. 量化原则

量化必须真实。没有真实数据时，只能写“建议指标”或“可补充指标”，并提醒用户确认后再写入正式简历。

可量化维度：

- **检索质量**：Top-K 命中率、引用覆盖率、无证据回答拦截率、答案可追溯比例。
- **响应性能**：平均响应耗时、P95 延迟、索引构建耗时、批量解析吞吐。
- **工程规模**：接入工具数量、索引文档数量、PDF/TXT 文件规模、核心模块数量、测试用例数量。
- **稳定性**：异常处理覆盖、重试机制、端到端测试通过率、解析失败率下降。
- **交付效率**：从需求到 Demo 的周期、自动化测试节省时间、配置向导减少部署步骤。

写作时可以表达为：“建议补充：Top-K 命中率、P95 响应耗时、索引文档数量，需由用户确认真实数据后使用。”

## 4. 技术标签

技术标签要按岗位方向选择，避免把没有实际使用的技术写进技术栏。

- **RAG**：Retrieval-Augmented Generation、向量/关键词检索、文档切分、证据合并、引用追踪、可追溯来源、招股书 PDF/TXT 上传与索引。
- **Agent**：工具调用、任务编排、MCP 标准工具服务、多步骤推理、证据校验、结果约束。
- **Backend**：Python、FastAPI、SQL、SQLite/PostgreSQL、REST API、配置管理、测试与日志。
- **Financial AI**：本地优先金融问答、财务指标查询、招股书分析、结构化财务数据、SQL-first evidence path。
- **Frontend Demo**：Streamlit/React、问答界面、文件上传、检索证据展示、配置向导、结果可视化。

## 5. 诚实包装边界

允许强调的能力：

- 本地优先金融问答。
- SQL-first evidence path。
- 招股书 PDF/TXT 上传与索引。
- MCP 标准工具服务。
- 证据合并、校验与可追溯来源。

禁止声称的能力：

- 已上线生产 SaaS。
- OAuth/JWT 认证体系。
- 多租户系统。
- 云托管生产部署。
- CLIP/FAISS。
- 直接由 LLM 生成 SQL。
- 伪造真实业务指标。

如果用户希望“包装得更强”，应优先加强问题定义、工程细节、指标建议和岗位匹配，而不是虚构生产能力或不存在的技术栈。

## 6. 输出前检查

输出简历项目前，逐项检查：

- 是否包含背景 -> 目标 -> 过程 -> 结果。
- bullet 是否以主动动词开头。
- 是否明确个人贡献，而不是只写团队成果。
- 是否至少包含 3 个量化点或建议量化点。
- 是否没有虚构生产 SaaS、认证、多租户、云部署等能力。
- 技术栏是否匹配目标岗位，且只包含项目中真实使用或可解释的技术。
